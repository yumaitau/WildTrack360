## Summary

**Goal:** Produce a complete, browsable OpenAPI 3.1 description of all **116 routes** under `src/app/api/`, committed as static artifacts (`docs/api/openapi.json` + a fully-offline Redoc `index.html`), generated by `zod-openapi` v4 from authored doc-only zod schemas, with a new GitHub Actions workflow guaranteeing the route map and status codes can never silently drift.

## Context

WildTrack360 exposes 116 route files (~175 handlers) across ~30 domains, but the only written description is the RBAC/species tables in `README.md` and prose in `.product/FEATURES.md` — not a machine-readable, per-endpoint contract. This task delivers that contract for internal developers (the immediate audience) and lays the foundation for opening the deliberately external-facing surfaces (`public/*` checkout, `portal/*`, Square webhook) to third parties later. Only 4 of 116 routes use zod today, so request/response **bodies are author-and-review-verified against the handlers**, not machine-derived — an honest boundary the CI guards reflect (they assert the route map + status codes, not body fidelity). Full requirements: `docs/prd/2026-06-17-api-documentation-openapi.md`.

**Two decisions postdate the PRD** and are recorded here so the plan doesn't read as drift against it: (1) a **new GitHub Actions workflow** is created — the repo has no `.github/workflows/` today, so `api-docs.yml` is its first CI workflow; (2) the renderer is **Redoc offline single-file**, not the PRD's Scalar default — the user overrode the default in favour of the PRD's own pre-authorized Redoc fallback (PRD Key Decisions, renderer row), so the committed `index.html` renders with zero network. The PRD's remaining "Scalar" references are superseded by decision (2).

## Out of Scope

## Approach

**Chosen:** `zod-openapi` v4 (`createDocument`) assembling **standalone doc-only zod schemas** authored in a new `src/lib/openapi/` tree; rendered to a self-contained offline HTML by `@redocly/cli build-docs`; guarded by a new `.github/workflows/api-docs.yml`.

**Why:** `next-openapi-gen` needs Node ≥24 (project is Node 22); `zod-openapi@4.2.4` is zod-3 / Node-18+ compatible and runs as-is (peer `zod ^3.21.4` vs the project's `^3.25.76`). Authoring schemas standalone — *not* importing the handlers — keeps the generation graph free of `server-only` (every handler transitively imports `clerk-server`/`rbac`/`prisma`, all marked `'server-only'`) and avoids booting Prisma/Clerk at generate time, at the cost of body fidelity being a reviewer responsibility rather than CI-asserted. Redoc's offline single-file (user's choice over Scalar's CDN render) means the committed `index.html` renders with zero network.

## Context for Implementer

Four cross-task invariants every task must respect:

1. **Generation graph must stay `server-only`-free.** `scripts/generate-openapi.ts` and everything under `src/lib/openapi/` import only `zod` + `zod-openapi` — **never** a `route.ts`, `@/lib/prisma`, `@/lib/clerk-server`, or `@/lib/rbac`. (`rbac.ts:1` and `clerk-server.ts:1` literally begin with `'server-only';`; `prisma.ts:1` does **not** — it's `import { PrismaClient }` — but importing it instantiates a `PrismaClient`/pulls `@prisma/client`, equally fatal at generate time. The rule is identical: import none of them.) Response schemas *mirror* Prisma model shapes by hand; they do not import `@prisma/client` types at runtime (a `type`-only import for authoring is fine since it erases). Where a route already has a zod schema (`report-queries` create, `wally`), mirror it 1:1 into the openapi module rather than importing the route file. This invariant is mechanically enforced by a structural-guard test in Task 3 (a careless `import` in any fan-out task crashes the generate under `npm ci` — every other server-touching script in `package.json` runs `tsx --tsconfig tsconfig.scripts.json` precisely because plain `tsx` + `server-only` crashes).
2. **Status extraction is PER-METHOD, and implicit statuses are real.** The extractor MUST slice each *exported handler function body* (`GET`/`POST`/`PATCH`/`PUT`/`DELETE`) and attribute statuses to that method only — **never** a file-level union. 46 of the 116 route files export ≥2 methods; e.g. in `src/app/api/animals/route.ts`, `GET` returns 200/400/401/403/500 while `POST` adds 201/422 — a file-level union would wrongly demand 201/422 on `GET`. Within each handler slice, treat as **implicit**: a `NextResponse.json(...)` / `new NextResponse(...)` return with no explicit `status:` ⇒ **200**; a `NextResponse.redirect(...)` with no explicit status ⇒ **307** (the Next default — `src/app/api/square/oauth/authorize/route.ts:23`; verify against the installed Next version, do **not** assume 302). CSV/binary exports like `members/export` carry an *explicit* `status: 200` literal — not implicit; use a bare-`NextResponse.json` GET (`animals`) as the implicit-200 fixture, not the export.
3. **Non-literal statuses defeat a naive `status: NNN` scan — three escape hatches.** A plain literal regex misses real codes, so the extractor combines: **(a) all literals in a status expression** — scan each status-bearing return for *every* `[1-5]\d\d` literal, so a ternary like `status: smsResult.blocked ? 422 : 500` (`src/app/api/pindrop/route.ts:71`) yields both 422 and 500. **(b) A response-helper allowlist** for statuses not visible locally: `gateFeature(...)`→**404** (returns a `NextResponse` directly — `src/lib/features.ts:51-54`, returned by gated routes via `if (gated) return gated;`, `src/app/api/members/export/route.ts:38-39`), and numeric-arg helpers like `errorResponse(msg, 502)` (`src/app/api/weather/route.ts:35-38`) where the status is the literal **arg**. **(c) A tested per-route override map** (`route-status-overrides.ts`) for *fully dynamic* statuses no scan can resolve — e.g. `status: error.status` backed by `class ReportAccessError(public status: number)` (`src/lib/custom-query/access.ts:22`), where the author records the enumerated set (`{400,401}`) for each affected route. Helpers that THROW and are caught into a local `status:` literal (e.g. `requirePermission`→403, `members/export:41-44`) are already visible — no special handling. The **same** `extractStatuses()` feeds the coverage gate, so anything it can't resolve via (a)/(b)/(c) is a silent under-assert; a Task-3 test asserts the override map is non-empty for every route whose only dynamic status source is `error.status`.
4. **Determinism of `openapi.json`.** The generated JSON must be byte-identical across machines/runs (the drift-guard diffs it): serialise with recursively-sorted object keys and a trailing newline. Pin every generation dep with `--save-exact` so a transitive patch bump can't flip the drift-guard under `npm ci`.

**Fan-out DoD convention (Tasks 4–10):** `coverage.test.ts` asserts only route+method presence and status-code presence — it does **not** verify min-role/permission accuracy or response-body fidelity, which stay reviewer-verified (the honest boundary above). A green coverage run is therefore *necessary but not sufficient* for a batch; each fan-out DoD below separates the **CI-asserted** criteria (route/method/status present) from the **reviewer-asserted** criteria (role/permission correct, body schema faithful). Do not read a green gate as proof the schemas are correct.

## Runtime Environment

No running service required — the deliverable is static. The only commands are:

- **Drift check:** `npm run docs:api:check` → `scripts/check-openapi-drift.ts` regenerates the document in-memory (sort-keys + trailing newline) and string-compares it to the committed `docs/api/openapi.json` (exit 1 on diff; never overwrites it). - **Coverage tests:** `npx vitest run src/lib/openapi` (no DB needed — filesystem scan + JSON parse).

## Assumptions

- All 116 handlers export HTTP methods as `export async function GET|POST|PATCH|PUT|DELETE` (verified on the sampled handlers). The route-surface extractor still also matches `export const GET =` / `export function GET` for robustness — Tasks 3 onward depend on this enumeration being complete.
- The error envelope is uniformly `{ error: string }` for 4xx/5xx (confirmed in `animals`, `report-queries`, `public/checkout/donation`, `square/webhook`). Non-`{error}` shapes, if any surface during fan-out, get a per-route response schema — Tasks 4–10 depend on this default holding for the majority.
- `package-lock.json` is committed (it is), so CI `npm ci` is valid.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Status-code extractor under-asserts: file-level (misses the per-method split), or skips implicit 200/307, ternary/helper-arg/fully-dynamic codes | Medium | High | Per-method extraction (slice each handler body); implicit 200 (`NextResponse.json`) + 307 (`redirect`); all-literals-in-expression scan (ternaries); response-helper allowlist (`gateFeature`→404, `errorResponse(_,NNN)`); tested per-route override map for `error.status` (invariants #2–#3); unit-tested against `animals` (per-method) + `members/export` + a ternary fixture; counts asserted in Task 3 |
| `openapi.json` non-deterministic key order ⇒ spurious CI drift failures | Medium | Medium | Recursive key-sort + trailing newline in the serializer; `--save-exact` deps + `npm ci` in CI |
| Generation script accidentally imports a `server-only` module and crashes under `tsx` | Medium | Medium | Invariant #1 above; generate script imports only `src/lib/openapi/`; a Task-1 DoD asserts `npm run docs:api` runs clean with no `server-only` error |
| Body fidelity drift (a handler's response shape changes, doc doesn't) — not CI-caught | High | Low | Stated boundary; mitigated by authoring each response schema beside its route's Prisma `select`/`include` for 1:1 review; reviewer/`/spec` responsibility, not overclaimed |
